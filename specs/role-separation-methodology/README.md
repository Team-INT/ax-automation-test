# 역할 분리 디자인 방법론 (Role Separation Design Methodology)

> **목적**: AI에게 한 방에 맡기지 않고, 카피-디자인-구현의 역할 분리를 통해 역대급 퀄리티의 웹사이트를 제작하는 체계적 워크플로우
>
> 적용 범위: 사이트 메인 페이지, 랜딩 페이지, 마케팅 페이지 등 콘텐츠 중심 페이지

## 1. 방법론 개요

### 1.1 문제 인식

AI에게 "웹사이트 만들어줘"라고 한 방에 맡기면:

- 카피가 뻔하다 ("혁신적인 솔루션", "다양한 서비스")
- 디자인이 천편일률적이다 (같은 레이아웃, 같은 색상)
- 콘텐츠와 디자인이 따로 논다 (메시지 누락, 톤 불일치)

### 1.2 해결 방향

**역할을 분리하여 각 단계별 최적의 AI 도구를 활용한다.**

```
[카피]              [디자인]             [구현]
Claude Code    →   Claude Code     →   Claude Code
(리서치+카피)       (디자인 스펙)         (코드 구현)
```

각 단계의 산출물이 다음 단계의 **입력이자 제약 조건**이 되어, 결과물의 일관성과 퀄리티를 보장한다.

### 1.3 적용 대상

| 대상 | 이 방법론 사용 | 기존 `/enf:design-feature` 사용 |
|------|:---:|:---:|
| 랜딩 페이지 | O | |
| 사이트 메인 페이지 | O | |
| 마케팅/캠페인 페이지 | O | |
| 제품 소개 페이지 | O | |
| Admin CRUD | | O |
| API 엔드포인트 | | O |
| 내부 대시보드 | | O |

## 2. 핵심 원칙

### 원칙 1: 카피부터 정리한다

> 카피가 흔들리면 화면도 흔들린다.

- 카피를 먼저 잡지 않으면 코드 작업 시간을 낭비하게 된다
- `copy.md` 파일이 디자인의 **설계도** 역할을 한다
- 뻔한 말을 **구체적인 숫자와 실제 상황**으로 바꾸는 것이 핵심

**나쁜 카피:**
```
"다양한 서비스를 제공합니다"
"시작하세요"
```

**좋은 카피:**
```
"월 평균 CS 문의 47건 → 12건으로 줄었습니다"
"30분이면 알 수 있습니다 — 무료 체험 시작"
```

### 원칙 2: 디자인 톤은 히어로 섹션에서 잠근다

> 첫 섹션이 사이트 전체의 톤을 결정한다.

- 폰트, 색상, 밀도, 레이아웃 등 디자인의 기본 톤을 히어로에서 정한다
- 나머지 섹션은 같은 톤으로 **체이닝** 방식으로 확장한다
- 페이지가 조각나는 것을 방지한다

### 원칙 3: AI에게 한 방에 맡기지 않는다

> 카피, 디자인, 코드 구현의 역할을 분리하여 각 단계별 최적의 AI 도구를 활용한다.

| 역할 | 도구 | 집중 영역 |
|------|------|-----------|
| 카피라이터 | Claude Code (`copy-strategist`) | 메시지, 톤, 설득력 |
| 크리에이티브 디렉터 | Claude Code (`design-director`) | 시각, 레이아웃, 인터랙션 |
| 프론트엔드 엔지니어 | Claude Code (`dev-assistant`) | 코드, 성능, 반응형 |

## 3. 3단계 워크플로우

### 3.1 전체 흐름

```
Step 1: 리서치 & 카피                    Step 2: 디자인                          Step 3: 구현
┌─────────────────────┐          ┌─────────────────────┐          ┌─────────────────────┐
│ /enf:copy-write     │          │ /enf:design-guide   │          │ /enf:design-implement│
│                     │          │                     │          │                     │
│ 1. 타겟 고객 분석    │  copy.md │ 1. 히어로 디자인 톤  │design-   │ 1. 디자인 시스템 구축 │
│ 2. 페인포인트 파악   │ ───────→│ 2. 섹션별 디자인     │spec.md  │ 2. 카피 교체         │
│ 3. 5섹션 카피 작성   │          │ 3. 디자인 토큰 추출  │ ───────→│ 3. 반응형 구현       │
│ 4. 구체적 수치 삽입  │          │ 4. Variant/Figma    │          │ 4. 브랜드 자산 적용  │
│                     │          │    작업 가이드       │          │                     │
└─────────┬───────────┘          └─────────┬───────────┘          └─────────┬───────────┘
          │                                │                                │
          ▼                                ▼                                ▼
   /enf:copy-review                 [사용자 디자인 검수]              /enf:code-review
   (품질 게이트 1)                  (품질 게이트 2)                  (품질 게이트 3)
```

### 3.2 Step 1: 리서치 & 카피 (Copy)

**담당**: `copy-strategist` 에이전트 / `/enf:copy-write` 커맨드

**프로세스:**

1. **타겟 고객 정의 및 문제점 파악**
   - 서비스의 이상적인 고객 페르소나 정의
   - 고객의 페인포인트, 검색 키워드, 원하는 결과 파악
   - 구조화된 JSON 형태로 리서치 결과 정리

2. **카피 방향 설정 및 초안 작성**
   - 리서치 결과를 바탕으로 5개 섹션 카피 작성:
     - **Hero**: 공감 질문 + 핵심 가치 제안
     - **Problem**: 문제 제기 + 구체적 수치
     - **Service**: 솔루션 소개 + 특징 목록
     - **Testimonials**: 고객 후기 + 구체적 경험
     - **CTA**: 허들을 낮추는 행동 유도

3. **카피 검수 및 구체화**
   - 뻔한 표현을 구체적 숫자/상황으로 교체
   - 각 섹션의 메시지 일관성 검증
   - `copy.md` 파일로 출력

**산출물**: `copy.md` ([템플릿 →](copy-template.md))

### 3.3 Step 2: 디자인 (Design)

**담당**: `design-director` 에이전트 / `/enf:design-guide` 커맨드

**프로세스:**

1. **copy.md 분석**
   - 각 섹션의 콘텐츠 구조 파악 (텍스트 양, 항목 수, 이미지 필요 여부)
   - 타겟 페르소나의 시각적 기대치 분석

2. **디자인 톤 결정**
   - 타겟 페르소나와 서비스 성격에 맞는 시각적 톤 설정
   - 히어로에서 컬러, 폰트, 밀도, 레이아웃 확정
   - 기존 디자인 레퍼런스가 있으면 해당 컨셉 우선 적용

3. **섹션별 디자인 스펙 작성**
   - 레이아웃, 배경, 컴포넌트, 인터랙션, Copy 매핑
   - ASCII 와이어프레임으로 레이아웃 시각화

4. **디자인 시스템 토큰 정의**
   - 컬러, 타이포, 간격 등 Tailwind `@theme` CSS 변수로 추출
   - `design-spec.md` 파일로 출력

> **참고:** Figma URL 제공 시 MCP 도구로 디자인 정보 추출 가능. 외부 도구(Variant 등)는 사용자가 명시적으로 요청한 경우에만 가이드 제공.

**산출물**: `design-spec.md` ([템플릿 →](design-spec-template.md))

### 3.4 Step 3: 구현 (Implementation)

**담당**: `dev-assistant` 에이전트 / `/enf:design-implement` 커맨드

**프로세스:**

1. **디자인 시스템 구축**
   - `design-spec.md`의 토큰을 Tailwind `@theme` 변수로 변환
   - 컬러, 폰트, 간격 등 CSS 변수 정의

2. **카피 교체**
   - Variant 내보내기 코드의 placeholder 텍스트를 `copy.md` 내용으로 교체
   - 카피의 구조(헤드라인, 서브카피, 본문)에 맞춰 컴포넌트 구성

3. **브랜드 자산 적용**
   - 이미지, 로고, 아이콘 교체
   - 브랜드 폰트 적용 (next/font)
   - 브랜드 컬러 확인

4. **반응형 구현**
   - Next.js + Tailwind CSS 기반
   - Mobile-first 반응형
   - 접근성 (WCAG 2.1 AA)

**산출물**: 구현된 코드 + Handoff Artifact

## 4. 역할 정의

### 4.1 copy-strategist (카피 전략가)

| 항목 | 내용 |
|------|------|
| **전문성** | 카피라이팅, 고객 심리, 마케팅 |
| **입력** | 서비스/제품 설명, 타겟 고객 정보 |
| **출력** | `copy.md` |
| **도구** | Read, Glob, Grep (읽기 전용) |
| **제약** | 코드를 작성하지 않음, 디자인 지시를 하지 않음 |
| **페르소나 렌즈** | "이 메시지가 타겟에게 설득력 있는가?" |

### 4.2 design-director (디자인 디렉터)

| 항목 | 내용 |
|------|------|
| **전문성** | 시각 디자인, UI/UX, 디자인 시스템 |
| **입력** | `copy.md` + 브랜드 가이드라인 |
| **출력** | `design-spec.md` |
| **도구** | Read, Glob, Grep, Figma MCP 도구 |
| **제약** | 카피를 수정하지 않음, 코드를 작성하지 않음 |
| **페르소나 렌즈** | "시각적으로 신뢰감/매력을 어떻게 전달하는가?" |

### 4.3 dev-assistant (구현 엔지니어) — 기존 에이전트 활용

| 항목 | 내용 |
|------|------|
| **전문성** | 프론트엔드 개발, 성능 최적화 |
| **입력** | `copy.md` + `design-spec.md` + 디자인 코드 |
| **출력** | 구현된 코드 |
| **도구** | Read, Edit, Write, Glob, Grep, Bash |
| **제약** | 카피를 수정하지 않음, 디자인 판단을 하지 않음 |
| **페르소나 렌즈** | "성능이 타겟의 기기 환경을 고려하는가?" |

## 5. 산출물 (Artifacts)

### 5.1 copy.md

각 단계에서 생성되는 핵심 산출물. 상세 템플릿은 [copy-template.md](copy-template.md) 참조.

**구조 요약:**

```
# Copy: {프로젝트명}
## 고객 리서치 (JSON)
## 섹션 1: Hero
## 섹션 2: Problem
## 섹션 3: Service
## 섹션 4: Testimonials
## 섹션 5: CTA
```

### 5.2 design-spec.md

디자인 단계의 산출물. 상세 템플릿은 [design-spec-template.md](design-spec-template.md) 참조.

**구조 요약:**

```
# Design Spec: {프로젝트명}

## 외부 도구 산출물
## 디자인 시스템 토큰
```

## 6. 품질 게이트

각 단계 전환 시 품질을 검증한다. 상세 기준은 [quality-gates.md](quality-gates.md) 참조.

### Gate 1: 카피 검증 (Step 1 → Step 2)

`/enf:copy-review` 커맨드로 실행.

| 검증 항목 | 통과 기준 | 실패 조건 |
|----------|----------|----------|
| 구체성 | 모든 섹션에 구체적 숫자/데이터 포함 | "많은", "다양한" 같은 추상적 표현 |
| 완전성 | 5개 섹션 모두 작성 | 누락된 섹션 존재 |
| 페르소나 정합성 | 타겟 고객의 페인포인트 직접 언급 | 범용적 메시지 |
| CTA 명확성 | 구체적 행동 동사 + 허들 낮추기 | 모호한 CTA |
| 톤 일관성 | 전 섹션 동일 톤 유지 | 섹션 간 톤 변화 |

### Gate 2: 디자인 검증 (Step 2 → Step 3)

사용자가 직접 검수.

| 검증 항목 | 통과 기준 | 실패 조건 |
|----------|----------|----------|
| 톤 잠금 | 모든 섹션이 히어로의 디자인 톤과 일치 | 스타일 드리프트 |
| 카피 무결성 | `copy.md` 원문 그대로 사용 | AI가 카피를 임의로 변경 |
| 반응형 계획 | 모바일/태블릿 브레이크포인트 명시 | 데스크탑 전용 |
| 토큰 추출 | 컬러/폰트/간격이 변수로 문서화 | 하드코딩된 값만 존재 |

### Gate 3: 구현 검증 (Step 3 완료)

`/enf:code-review` + `/enf:perf-audit` 커맨드로 실행.

## 7. enf 플러그인 통합

### 7.1 새로운 커맨드

```
/enf:copy-write "타겟: ..."        ← Step 1 진입점
/enf:copy-review                    ← Gate 1
/enf:design-guide                   ← Step 2 진입점
/enf:design-implement               ← Step 3 진입점
```

### 7.2 새로운 에이전트

```
@copy-strategist                    ← 카피 전문가
@design-director                    ← 디자인 디렉터
```

### 7.3 새로운 스킬

```
role-separation-workflow            ← 트리거: 카피, 역할 분리, Variant, 랜딩페이지 등
```

### 7.4 전체 워크플로우

```
/enf:task "랜딩페이지 제작"
       ↓
/enf:copy-write "타겟: ..."        ← Step 1: 카피 (copy-strategist)
       ↓ [copy.md 생성]
/enf:copy-review                    ← Gate 1: 카피 검증
       ↓ (pass)
/enf:design-guide                   ← Step 2: 디자인 (design-director)
       ↓ [design-spec.md 생성]
/enf:design-implement               ← Step 3: 구현 (dev-assistant)
       ↓ [코드 생성]
/enf:code-review → /enf:commit → /enf:push → /enf:pr
```

## 8. 기존 워크플로우와의 관계

### 8.1 분기 기준

```
/enf:design-feature 실행 시:

  ┌─ 사이트/랜딩페이지/마케팅?  ──→  역할 분리 워크플로우 안내
  │                                  (/enf:copy-write 부터 시작 권장)
  │
  └─ Admin/CRUD/API?           ──→  기존 architecture-expert 워크플로우
                                     (Route → Schema → Component 설계)
```

### 8.2 호환성

- 기존 `/enf:design-feature` 워크플로우는 **그대로 유지**
- 역할 분리 워크플로우는 **추가 경로**로 제공
- 각 커맨드는 **독립 실행 가능** (전체 워크플로우를 반드시 따를 필요 없음)
- 예: 카피가 이미 있으면 `/enf:design-guide`부터 시작 가능
